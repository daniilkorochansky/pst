# PST — Pawn Standards and Techniques

PST — это свод сообщественных стандартов, соглашений и передовых практик
по развитию Pawn в проектах open.mp и SA-MP.

Цель PST заключается в повышении читаемости кода, удобства его сопровождения,
совместимости с инструментами и улучшении организации проекта.

## 2. Типы документов PST

Все документы PST делятся на три категории:

1. **Standards (Стандарты):** Документы, описывающие конкретные технические правила написания кода, архитектуры модов, интеграции баз данных и использования инструментов (компиляторов, инклудов).
2. **Informational (Информационные):** Руководства, туториалы, обзоры лучших практик и рекомендации. Они не устанавливают жестких правил, но помогают сообществу развиваться.
3. **Process (Процессы):** Документы, описывающие внутренние правила самого проекта PST.

## 3. Жизненный цикл и статусы PST

Каждое предложение проходит строгий процесс отбора и обсуждения. Статус документа отображается в его заголовке:

* **Draft (Черновик):** Начальная стадия. Документ открыт для предложений, критики и активного редактирования в ветке Pull Request.
* **Review (На рецензии):** Текст документа стабилизирован. Сообщество проводит финальное техническое ревью.
* **Accepted (Принято):** Стандарт официально утвержден и добавлен в основную ветку (`main`) репозитория. Он рекомендуется к внедрению во все современные проекты.
* **Rejected (Отклонено):** Предложение признано неэффективным, устаревшим или вредным. Документ отправляется в архив с обязательным указанием причин отказа.
* **Obsolete (Устарело):** Ранее принятый стандарт, который потерял актуальность (например, из-за глобальных обновлений open.mp) и был заменен новым PST.

## 4. Как предложить свой PST: Пошаговый процесс

Стать автором стандарта может любой участник сообщества. Процесс выглядит следующим образом:

1. **Обсуждение идеи:** Перед написанием текста создайте тему в разделе `Issues` репозитория PS. Опишите проблему, которую вы хотите решить.
2. **Создание черновика:** Если идея поддержана, сделайте форк репозитория PST и создайте новый файл по [ШАБЛОНУ](https://github.com/daniilkorochansky/pst/blob/main/pst-template.md) (например, `pst-0001-code-style-guide.md`) в статусе `Draft`.
3. **Открытие Pull Request (PR):** Отправьте ваш черновик на рассмотрение. Начнется этап публичного обсуждения, где каждый пункт будет разбираться в комментариях к коду.
4. **Доработка:** Автор обязан отвечать на конструктивную критику и вносить правки в PR.
5. **Фиксация решения:** Редактор PST (или совет экспертов) на основе консенсуса принимает решение о переводе документа в статус `Accepted` или `Rejected`.

## 5. Шаблон заголовка (Metadata)

Каждое новое предложение должно начинаться со следующего блока метаданных:

```text
PST: [Номер документа, выдается редактором]
Title: [Лаконичное название стандарта]
Author: [Имя/Ник] <[Email или ссылка на профиль]>
Status: [Draft / Review / Accepted / Rejected / Obsolete]
Type: [Standards Track / Informational / Process]
Created: [Дата в формате DD-Mmm-YYYY, например 20-Jun-2026]
```
